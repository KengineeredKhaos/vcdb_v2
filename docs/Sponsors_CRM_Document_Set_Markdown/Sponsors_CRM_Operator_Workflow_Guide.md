# Sponsors CRM — Operator Workflow Guide

## Purpose

This guide explains how an operator should use Sponsors CRM in the app as
it exists now.

It is written from a practical, day-to-day workflow perspective rather
than a technical perspective.

## Operator mindset

Use Sponsors CRM as:
- a relationship memory aid
- a cultivation work organizer
- a next-action support tool

Do **not** use it as:
- an automatic truth machine
- a replacement for phone calls, judgment, or local knowledge
- a rigid sales pipeline

The system is there to help you remember, review, and follow through.

## Core operator tasks

The current workflow supports five main tasks:
1. review sponsor dossier information
2. review funding opportunity matches
3. create cultivation tasks
4. review cultivation outcomes
5. create follow-up tasks or preserve sponsor knowledge

## Workflow 1 — Review a sponsor dossier

Use the sponsor detail page when you need to understand a sponsor as a
whole.

Look for:
- CRM posture and factors
- profile note hints
- recent cultivation activity
- latest cultivation touch
- operator workflow actions
- relationship notes already preserved

Questions to answer:
- what do we know about this sponsor?
- what style of ask seems appropriate?
- have we contacted them recently?
- is there an obvious next action?

## Workflow 2 — Review a funding opportunity

Use the funding opportunity detail page when working from a specific
need or ask.

The page should help you answer:
- which sponsors look promising for this demand?
- what are the reasons or cautions?
- who has already been contacted about this demand?
- did the last touch surface interest?
- is follow-up pending or already scheduled?

The page is demand-centered, not sponsor-centered.

## Workflow 3 — Create a cultivation task

Create a cultivation task when you want to initiate a sponsor touch.

Typical reasons:
- a sponsor appears to fit a current demand
- a sponsor has not been contacted in a while
- an operator wants to test interest or gather more information
- a profile note suggests a useful contact path

When creating a cultivation task:
- make sure the sponsor is correct
- carry the demand context when relevant
- keep the task title and purpose readable
- use the task as explicit work, not as a hidden reminder to yourself

## Workflow 4 — Review cultivation outcomes

Once cultivation work has happened, Calendar returns outcome signals to
Sponsors.

Review the returned outcome row for:
- task title
- status
- due or completion timing
- linked funding demand, if present
- outcome note
- follow-up recommended
- off-cadence follow-up signal
- funding interest surfaced

Treat these as **operator cues**, not commands.

## Workflow 5 — Create a follow-up task

Create a follow-up task when the completed outcome suggests another
touch should happen.

Typical indicators:
- follow-up recommended
- off-cadence follow-up flagged
- funding interest surfaced
- operator judgment says the relationship should be advanced

The follow-up task should preserve:
- sponsor identity
- funding demand linkage, if present
- a compact summary of the prior touch

The goal is continuity, not automation.

## Workflow 6 — Promote an outcome note into relationship knowledge

Promotion should be used when an outcome note contains sponsor knowledge
that should survive beyond the individual task.

Examples:
- sponsor prefers email first
- sponsor wants concise asks
- sponsor is interested in veteran-only efforts
- sponsor asked to revisit after a certain season
- sponsor is open but needs board review

Promotion is idempotent. Repeating the same promotion should not create
duplicate relationship knowledge.

## Queue-based interpretation of sponsor “temperature”

VCDB does not use an automated sponsor-temperature score.

Instead, operators should determine sponsor temperature from the queue of
available information:
- recent contact frequency
- last cultivation outcome
- funding interest surfaced or not
- follow-up pending or scheduled
- relationship notes
- operator memory and local context

In plain terms:
- a sponsor with a recent warm note and pending follow-up is “warm”
- a sponsor with old activity and little information is “cold”
- a sponsor with a recent decline or sensitivity note may need space

The app should support that judgment, not attempt to replace it.

## Good operator habits

### Do
- keep outcome notes concise and useful
- create explicit follow-up tasks when needed
- promote durable relationship knowledge
- review recent demand-centered activity before making a new touch
- treat cues as prompts for judgment

### Do not
- assume the system knows more than local knowledge
- create duplicate tasks without purpose
- use CRM factors as rigid truth
- overfit the workflow to corporate CRM language
- confuse a signal with a decision

## Present limitations

At the current stage:
- sponsor temperature is not system-derived
- relationship stage is not formalized
- next-touch summary is still emerging
- queue and reporting views are still limited
- stewardship tracking is not yet a full subsystem

These are acceptable limits for the present MVP.

## Operator success test

An operator should be able to:
- open a funding opportunity
- identify likely sponsors
- see whether someone already made contact
- decide whether to create or schedule a follow-up
- open the sponsor dossier
- preserve important relationship knowledge

without needing a separate spreadsheet or personal memory system.
